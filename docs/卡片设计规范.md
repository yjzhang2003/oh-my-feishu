# 卡片设计规范

本文档记录 oh-my-feishu 卡片的视觉与实现约定，避免后续改动破坏 CardKit 流式更新或交互回调。

## 总体原则

- 优先使用卡片 JSON 2.0 组件，保持 `update_multi: true`。
- 组件 `element_id` 使用短 ID，控制在 20 字符内。
- 标题区使用 `icon`、`subtitle`、`text_tag_list` 建立状态和上下文。
- 内容区优先用结构化组件表达层级，不用长段无分隔 Markdown。
- 不为了视觉效果引入复杂嵌套。容器嵌套过深会影响移动端可读性。

## 菜单卡片

菜单卡片是静态交互卡片，可以使用较丰富的布局组件：

- 使用 `column_set` 做左右入口布局，`flex_mode` 优先用 `bisect`，窄屏下仍有稳定表现。
- 使用 `interactive_container` 承载可点击入口，比单独按钮更容易表达“标题 + 说明 + 图标”。
- 使用 `table` 展示固定命令清单，避免命令说明堆成长列表。
- 使用 `standard_icon` 增强识别，例如 `chatbox_outlined`、`add_outlined`、`history_outlined`、`folder_outlined`。
- 不使用 `collapsible_panel` 做主导航。菜单需要一眼可见，折叠面板会增加操作成本。

## 流式回复卡片

流式回复卡片的核心是稳定追加和更新文本，视觉增强必须保守：

- 卡片初始 body 只放轻量上下文说明和分割线，实际回答仍通过 `addCardElements` 追加 `markdown`。
- 正文流式组件只使用可被 `/content` 接口更新的 `markdown`，不要包进复杂容器。
- 思考内容可以使用 `collapsible_panel`，默认收起，作为次要信息承载。
- 不在流式正文区域使用 `interactive_container`、`table`、复杂 `column_set`。这些组件不适合持续文本 PUT 更新。
- 不关闭 `streaming_mode`。当前实测关闭会触发飞书重渲染，可能导致流式内容消失。
- 完成态只更新 header 的 `status_tag` 和 summary，避免全量重写卡片。

## 视觉语言

- 直接对话使用 `turquoise`，图标优先 `chatbox_outlined`。
- 目录会话使用 `indigo`，图标优先 `folder_outlined`。
- 处理中状态使用橙色标签，完成状态使用绿色标签。
- thinking/日志类内容使用灰色弱化，避免抢正文优先级。

## 修改检查清单

- `npm run build`
- `npm test -- --run`
- 检查新图标 token 是否存在于 `docs/图标库.md`
- 确认流式卡片仍只对 `markdown` 组件调用 `updateCardContent`
- 确认卡片 JSON 大小未接近 30KB 限制
